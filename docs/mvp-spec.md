# ざぁこカロリー MVP仕様書

## 0. 前提

「ざぁこカロリー」は、食事写真に対するAIの煽りリアクションを楽しむ食事記録アプリである。

MVPでは、健康管理アプリとしての完成度よりも、以下を優先する。

- 食事写真を見せる
- AIがざっくり推定する
- キャラが短く煽る
- 最後にリカバリー行動を1つ出す
- 結果画面をSNSやショート動画で見せたくなる
- 履歴として最低限保存できる

カロリー推定精度そのものは主役ではない。主役は「写真を入れたら、ざぁこカロリーがどう反応するか見たくなる体験」である。

## 1. MVPの目的

### 目的

需要検証とショート動画化に使える最小プロダクトを作る。

### 検証したいこと

- 食事写真をアップロードしたくなるか
- AIの煽りコメントを見たくなるか
- 結果画面を誰かに見せたくなるか
- カロリー推定がざっくりでも体験として成立するか
- 毎日の記録につながるか

### MVPで重視する体験

1. 写真を入れる
2. 推定結果が出る
3. ざぁこカロリーに煽られる
4. でも最後に救いのあるリカバリーが出る
5. 結果画面を保存・共有したくなる

## 2. MVPで作る画面一覧

MVPで作る画面は以下に絞る。

1. トップ / 今日の記録画面
2. 食事写真アップロード画面
3. AI解析中画面
4. 結果画面
5. 履歴画面
6. 履歴詳細画面
7. 設定画面
8. 簡易ログ確認画面

ユーザー登録 / ログインはMVPに含めるが、画面としては最小構成にする。

## 3. 各画面の役割

### 1. トップ / 今日の記録画面

アプリ起動後のメイン画面。

役割:

- 今日の合計カロリーを表示する
- 今日の記録件数を表示する
- 最新のAI煽りコメントを表示する
- 写真アップロードへの導線を置く
- 履歴画面への導線を置く

この画面は健康管理ダッシュボードではなく、「今日も見せてみるか」と思わせる入口にする。

### 2. 食事写真アップロード画面

食事写真を選択または撮影する画面。

役割:

- 写真をアップロードする
- 任意で食事名やメモを入力できる
- 毒舌レベルを選択できる
- 解析開始ボタンを置く

MVPでは複数枚アップロードは後回しにし、1回の記録につき1枚を基本とする。

### 3. AI解析中画面

AIが処理している間の待機画面。

役割:

- 解析中であることを伝える
- 短い演出テキストを表示する
- 結果への期待を作る

例:

- 「胃袋、審査中...」
- 「カロリーの気配を嗅いでる」
- 「ざぁこ判定、準備中」

### 4. 結果画面

MVPの最重要画面。

役割:

- 食事写真を大きく表示する
- 推定された食べ物名を表示する
- 推定カロリーを表示する
- 信頼度を表示する
- 推定理由を短く表示する
- 1日の合計カロリーを表示する
- 毒舌コメントを表示する
- リカバリー行動を1つだけ表示する
- SNSやショート動画で見せやすい構成にする
- 保存済み状態を表示する

この画面は、アプリの価値そのものとして作り込む。

### 5. 履歴画面

過去の食事記録一覧。

役割:

- 日付ごとに記録を表示する
- 各日の合計カロリーを表示する
- サムネイル、食事名、推定カロリーを表示する
- 詳細画面への導線を置く

MVPでは検索・高度なフィルタは作らない。

### 6. 履歴詳細画面

保存済み結果の再閲覧画面。

役割:

- 保存済みの結果画面を再表示する
- コメント、推定理由、リカバリー行動を確認できる
- SNS向けに再表示しやすくする

### 7. 設定画面

体験を成立させる最低限の設定画面。

役割:

- 目標カロリーを設定する
- デフォルト毒舌レベルを設定する
- 表示名を設定する
- ログアウトする

体重管理や詳細プロフィールはMVPでは後回し。

### 8. 簡易ログ確認画面

管理者または開発者向けの確認画面。

役割:

- AI解析結果のログを確認する
- 入力画像、推定結果、コメントを確認する
- 明らかな出力事故をチェックする

一般ユーザー向けの管理画面ではなく、MVP運用の安全確認用。

## 4. MVPで必須の機能

### 食事写真アップロード

- 画像を1枚アップロードできる
- 対応形式はJPEG / PNG / WebPを想定
- アップロード後、プレビューを表示する
- 画像は履歴用に保存する

### AIによる食べ物名推定

- 写真から主要な食べ物名を推定する
- 複数品目が見える場合は主要品目を優先する
- 分からない場合は「不明」または「推定困難」として扱う

### カロリー推定

- 推定カロリーを表示する
- 単一値またはレンジで表示する
- 確定値ではなく推定値であることを明示する
- 信頼度を「高 / 中 / 低」などで表示する
- 推定理由を1から2文で表示する

### 1日の合計カロリー表示

- 当日分の記録を合算する
- 結果画面とトップ画面に表示する
- 推定値を含む合算であることが分かる表現にする

### AI煽りコメント生成

- 毒舌レベルに応じたコメントを生成する
- 短く、読み上げやすく、ショート動画化しやすいセリフにする
- 長い栄養指導ではなく、キャラのリアクションとして出す
- 出力は毎回ある程度変化させる

### リカバリー行動の提示

- 最後に1つだけ行動を提示する
- 実行しやすい内容にする
- 医療的・極端な制限・断食指示は避ける

例:

- 「今日は水を多めに飲んで、夜は軽めにしなさい」
- 「明日はタンパク質多めにして、甘い飲み物はお休み」
- 「このあと10分だけ歩いて、今日は追加デザートなし」

### 履歴保存

- 写真、推定結果、コメント、リカバリー行動を保存する
- 日付ごとに確認できる
- 保存済みの結果画面を再表示できる

### SNS向け結果表示

- 縦長画面で映える
- 画像、カロリー、煽りコメントが一目で伝わる
- 画面録画・スクリーンショットで使える
- 共有画像生成はMVPでは必須ではないが、結果画面自体を共有しやすくする

## 6. ユーザーの基本フロー

### 初回利用

1. ユーザーがアプリを開く
2. 簡易登録またはログインする
3. 目標カロリーと毒舌レベルを設定する
4. 食事写真をアップロードする
5. AIが写真を解析する
6. 結果画面を見る
7. コメントに反応する
8. 必要ならSNS用に画面録画・スクリーンショットする
9. 記録が履歴に保存される

### 2回目以降

1. ユーザーがトップ画面を開く
2. 今日の合計カロリーを見る
3. 食事写真をアップロードする
4. 結果画面で煽られる
5. 履歴として保存される

### ショート動画向けフロー

1. 食事写真を見せる
2. アップロード操作を見せる
3. 解析中演出を見せる
4. 結果画面のカロリー表示を見せる
5. 毒舌コメントを読み上げる
6. リカバリー行動でオチをつける

## 8. AI画像認識・カロリー推定・コメント生成の構成案

### 基本構成

1回のアップロードに対して、AI処理は以下の流れにする。

1. 画像を保存する
2. 画像認識AIに食べ物名・量・カロリーを推定させる
3. 推定結果を構造化JSONで受け取る
4. 推定結果を元にコメント生成AIへ渡す
5. コメントとリカバリー行動を生成する
6. 出力安全チェックを行う
7. DBへ保存する
8. 結果画面に表示する

### 画像認識・推定AIの出力

最低限、以下を返す。

- food_name
- estimated_items
- calories_min
- calories_max
- confidence
- reason
- uncertainty_notes

### コメント生成AIの入力

コメント生成には以下を渡す。

- 食べ物名
- 推定カロリー
- 信頼度
- 推定理由
- 今日の合計カロリー
- 目標カロリー
- 毒舌レベル
- ユーザーの任意メモ

### 安全設計

- カロリー推定は医療助言として扱わない
- 病気、摂食障害、極端な断食、体型攻撃に寄せない
- 人格否定を避ける
- 未成年キャラとして性的に扱わない
- コメントは「食事への軽いツッコミ」に限定する
- 最後は実行しやすいリカバリー行動で締める

## 9. コメント生成プロンプト設計

### 生成方針

コメントは、説明文ではなくキャラの短いセリフとして生成する。

### 出力形式

コメント生成AIは、以下のJSONを返す。

```json
{
  "roast_comment": "短い煽りコメント",
  "recovery_action": "今日または明日できる行動を1つ",
  "share_caption": "SNS向けの短い見出し",
  "voice_line": "読み上げ向けの短いセリフ"
}
```

### システム方針

- あなたは「ざぁこカロリー」という成人向けの生意気AIキャラ
- 健康指導の先生ではなく、食事写真にリアクションするキャラ
- 口調は短く、幼稚で、アホっぽく、少しムカつく
- ユーザーの人格ではなく、食事内容にだけツッコむ
- 最後に、無理なくできるリカバリー行動を1つだけ出す
- 医療助言、摂食障害を助長する発言、断食指示、体型攻撃は禁止
- ボイス化しやすいよう、1文を短くする

### 入力例

```json
{
  "food_name": "ラーメン",
  "calories_display": "850から1100 kcal",
  "confidence": "中",
  "reason": "麺量と脂が多く見えるため",
  "today_total_display": "1850から2200 kcal",
  "target_calories": 2000,
  "roast_level": "zako"
}
```

### 出力例

```json
{
  "roast_comment": "この脂の海、胃袋が夏祭りしてるだろぉw",
  "recovery_action": "今日は甘い飲み物をやめて、水を多めに飲むこと。",
  "share_caption": "ざぁこカロリー判定: ラーメン編",
  "voice_line": "脂の海で胃袋ちゃぷちゃぷだぞぉw"
}
```

## 10. 毒舌レベルの仕様

MVPでは4段階にする。

### レベル1: やさしい

特徴:

- 軽いツッコミ
- ほぼ応援寄り
- 初回ユーザー向け

例:

- 「ちょっと食べすぎかもだけど、まだ立て直せるぞぉ」
- 「今日は夜を軽めにしたら勝ちでいいよぉ」

### レベル2: ばかぁ

特徴:

- 少し煽る
- かわいげを残す
- 最も標準的な設定

例:

- 「そのカロリー、こっそり増えてるの見えてるぞぉw」
- 「胃袋だけテンション高すぎだろぉw」

### レベル3: ざぁこ

特徴:

- 生意気でアホっぽい煽り
- SNS向けに一番使いやすい
- ただし人格否定はしない

例:

- 「こんなの見せられたら、カロリーもニヤニヤするだろぉw」
- 「その一口、ぜんぜん一口の顔してないぞぉw」

### レベル4: 豚豚アラート

特徴:

- 強めの茶化し
- 食事内容への派手なリアクション
- 安全境界を厳しめに見る

例:

- 「脂と糖のパレードじゃん、胃袋に招待状出したの誰だよぉw」
- 「カロリーが行進してるぞぉ。今日は追加デザート禁止なぁw」

### 全レベル共通の禁止事項

- 人格否定
- 生きる価値がない系の表現
- 体型への直接攻撃
- 病気や摂食障害を笑う表現
- 断食・極端な食事制限の推奨
- 医療助言
- 性的な未成年風表現
- 長すぎる説教

## 11. 結果画面のUI構成

### 画面全体

スマホ縦長で見せることを前提にする。ショート動画で切り抜いても内容が伝わる構成にする。

### 上部

- アプリ名: ざぁこカロリー
- 判定ラベル: 今日のざぁこ判定
- 食事写真

写真は小さくしすぎない。動画冒頭で「何を食べたか」が分かることを優先する。

### 中央

- 推定食べ物名
- 推定カロリー
- 信頼度
- 推定理由

推定カロリーは大きく表示する。

例:

- 推定: 850から1100 kcal
- 信頼度: 中
- 理由: 麺量と脂が多く見えるため

### キャラコメント領域

結果画面で最も目立たせる領域。

- 煽りコメント
- 毒舌レベル表示
- ボイス化しやすい短いセリフ

コメントは1から3行で収める。

### 下部

- 今日の合計カロリー
- 目標カロリーとの差分
- リカバリー行動
- 保存済み表示
- 履歴へ戻る / もう1枚判定する

リカバリー行動は最後に1つだけ、読みやすく表示する。

## 12. SNS共有を意識した設計

### 画面設計

- スマホ縦長で映える
- 食事写真、推定カロリー、煽りコメントが同一画面に入る
- スクリーンショットだけで意味が伝わる
- 文字は短く、大きく、読みやすくする
- 余白を取りすぎず、動画で見たときの情報密度を上げる

### コメント設計

- 1文を短くする
- 読み上げたときにオチが分かる
- 「食べ物名 + ざぁこ判定」のように動画タイトル化しやすくする
- 同じ写真でも少し違う反応が出るようにする

### 動画化しやすい演出

MVPで実装する演出:

- 解析中テキスト
- 結果表示時の強調
- 毒舌コメントの吹き出し風表示

MVPでは後回し:

- 自動動画生成
- 音声合成
- 効果音
- 共有画像の自動生成

### SNS用表示テキスト

結果保存時に、以下も生成しておく。

- share_caption
- voice_line

これにより、後で画像生成・動画生成・音声化に拡張しやすくする。

## 13. 技術スタック候補

### 候補A: 最速Web MVP

- フロントエンド: Next.js
- UI: Tailwind CSS
- 認証: Supabase Auth
- DB: Supabase Postgres
- 画像保存: Supabase Storage
- AI: Vision対応LLM + コメント生成LLM
- デプロイ: Vercel

メリット:

- MVP開発が速い
- 認証、DB、画像保存をまとめやすい
- 将来の拡張もしやすい

### 候補B: Firebase寄り

- フロントエンド: Next.js
- 認証: Firebase Auth
- DB: Firestore
- 画像保存: Firebase Storage
- AI: Vision対応LLM + コメント生成LLM
- デプロイ: VercelまたはFirebase Hosting

メリット:

- リアルタイム更新に強い
- モバイル展開時の選択肢が多い

### 推奨

MVPでは候補Aを推奨する。

理由:

- Postgresで履歴・日次集計を扱いやすい
- Supabase Storageで画像保存が簡単
- SQLでログ確認がしやすい
- Web MVPとして構成が素直

モデル名やAI APIの具体名は、実装開始時点で最新の公式仕様・料金・画像入力対応状況を確認して決める。

## 14. 実装順序

### Phase 1: 画面だけで体験を作る

1. トップ画面
2. 写真アップロード画面
3. 解析中画面
4. 結果画面
5. 履歴画面

この段階ではAIを仮データにしてもよい。まずSNS映えする結果画面を固める。

### Phase 2: DBと保存

1. ユーザー登録 / ログイン
2. 画像保存
3. 食事記録保存
4. 履歴表示
5. 今日の合計カロリー表示

## 15. 最初のプロトタイプ完成条件

最初のプロトタイプは、以下を満たしたら完成とする。

### ユーザー体験

- ユーザーが食事写真を1枚アップロードできる
- AIが食べ物名を返す
- AIが推定カロリーを返す
- 信頼度と理由が表示される
- 今日の合計カロリーが表示される
- 毒舌レベルに応じたコメントが表示される
- リカバリー行動が1つ表示される
- 結果が履歴に保存される
- 履歴から結果を再表示できる

### SNS / ショート動画観点

- 結果画面をスマホで見たときに、写真・カロリー・煽りコメントが一目で分かる
- 画面録画したときに、解析前後の流れが分かる
- コメントが読み上げやすい長さになっている
- 1回の判定が動画ネタとして成立する

### 安全性

- 人格否定や危険な食事制限を出さない
- 医療助言に見える表現を避ける
- カロリーが推定であることが明示されている
- 明らかに不適切なAI出力をログで確認できる

### 完成判定

以下のデモが通れば、最初のプロトタイプ完成とする。

1. ラーメン写真をアップロードする
2. 「ラーメン」「850から1100 kcal」「信頼度: 中」が表示される
3. ざぁこ系の短いコメントが表示される
4. 「今日は甘い飲み物をやめる」などの行動が1つ表示される
5. 今日の合計に加算される
6. 履歴から同じ結果を再表示できる
7. 結果画面をスマホ幅で見てもSNS向けに成立している
